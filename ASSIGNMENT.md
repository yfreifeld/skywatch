## Project Overview

SkyWatch is a distributed weather-query pipeline built end-to-end as a DevOps course project. Users submit a city name through a Flask frontend; the request is published to a RabbitMQ queue, picked up by a Python worker that calls the free [Open-Meteo](https://open-meteo.com) API, and the result is returned to the user. Any city in the world is supported — no API key required.

Every layer — from cloud provisioning to runtime observability — is managed as code and deployed through a fully automated GitOps pipeline.

### Application architecture

```
Browser
  │
  ▼
Flask Frontend (port 5000)
  │  publishes job
  ▼
RabbitMQ (AMQP 5672 / Management 15672 / Prometheus 15692)
  │  consumes job
  ▼
Python Worker (×2 replicas)
  │  calls
  ▼
Open-Meteo API (geocoding + forecast)
```

### Technology stack

| Layer | Technology | Purpose |
|---|---|---|
| Cloud | AWS EC2 t3.micro, eu-west-1 | 3-node cluster |
| Provisioning | Terraform | VPC, SG, EC2 instances, Ansible inventory |
| Configuration | Ansible | K3s install, kubeconfig, swap, kubectl alias |
| Kubernetes | K3s v1.29.5 + containerd | Lightweight K8s for t3.micro |
| Containerisation | Docker → GHCR | Multi-stage images: frontend + worker |
| CI/CD | GitHub Actions | Lint → Build → Push → tag-bump commit |
| Packaging | Helm 3 | skywatch chart (frontend, worker, RabbitMQ) |
| GitOps | ArgoCD v2.11.2 | Auto-sync from Git, prune + self-heal |
| Messaging | RabbitMQ 3.13-management | Decouples frontend from workers |
| Observability | kube-prometheus-stack | Prometheus scrape + Grafana dashboards |

### Node layout

| Node | Hostname | Workloads |
|---|---|---|
| Master | skywatch-master | K3s control plane only (tainted NoSchedule) |
| Worker 1 | skywatch-worker | ArgoCD (7 pods) |
| Worker 2 | skywatch-worker2 | skywatch app (frontend, workers, RabbitMQ), Prometheus, Grafana |

## Part 1 — Infrastructure Provisioning

### Terraform

A single `main.tf` declares the complete AWS footprint: the default VPC, a security group with the necessary rules (SSH, K3s API, NodePort range, intra-cluster), and one EC2 instance per role. The Ubuntu 22.04 AMI is looked up dynamically so the code stays current without hard-coding an AMI ID. After `terraform apply` the Ansible inventory is written automatically from a template.

| Setting | Value |
|---|---|
| Instance type | t3.micro (1 vCPU, 1 GB RAM) — AWS free-tier eligible |
| Region | eu-west-1 (Ireland) |
| Nodes | master · worker · worker2 |
| Key pair | skywatch-key → `~/.ssh/skywatch-key.pem` |
| State file | `terraform/terraform.tfstate` (gitignored) |
| Credentials | `terraform/terraform.tfvars` (gitignored) — never hard-coded |

> **Free-tier note:** AWS gives 750 EC2 hours/month. Three t3.micro instances running for a 10-hour session consume 30 hours — well within the limit. The cluster is destroyed after every session with `terraform destroy`.

### Ansible

Two roles provision the cluster. Both are **idempotent**: re-running the playbook on a live cluster only joins new nodes without disturbing existing ones.

#### `k3s_master` role

- Checks `systemctl is-active k3s` before attempting uninstall — skips reinstall on a healthy master.
- Installs K3s `v1.29.5+k3s1` pinned (v1.35 OOMs on t3.micro).
- Passes `--tls-san <public_ip>` so the TLS certificate includes the public IP as a SAN, enabling `kubectl` from a laptop.
- Disables Traefik (NodePort is used instead).
- Reads the join token and exposes it as an Ansible fact for the worker play.

#### `k3s_worker` role

- Checks `systemctl is-active k3s-agent` before uninstalling — prevents disrupting a running node.
- Joins using `K3S_URL` and `K3S_TOKEN` as **environment variables** to the installer pipe — the installer reads env vars, not CLI flags.
- Uses the **private IP** of the master for `K3S_URL`; public IPs route via the AWS IGW and bypass the security-group self-rule, making port 6443 appear closed.

### Issues encountered

| Symptom | Root cause | Fix |
|---|---|---|
| Worker couldn't reach master:6443 | AWS routes VPC→public IP via IGW, bypassing SG self-rule | Use private IP in `K3S_URL` |
| `kubectl` TLS cert error from laptop | Cert only included private IP SAN | `--tls-san <public_ip>` during install |
| K3s token not stored in agent env file | Token passed as CLI flag; installer only reads env var | `K3S_TOKEN=... sh -s - agent` |
| Re-running Ansible wiped live cluster | Uninstall task fired because uninstall script exists on healthy node | Added `systemctl is-active` check before uninstall |
| K3s v1.35 OOM on t3.micro | New API server + etcd footprint too large for 1 GB RAM | Pinned to `v1.29.5+k3s1` |

## Part 2 — Containerisation & CI/CD

### Application

The frontend is a Flask app; the worker is a plain Python process. Both have simple Dockerfiles using the official Python 3.11 slim base image. Images are pushed to **GitHub Container Registry (GHCR)**.

RabbitMQ uses the official `rabbitmq:3.13-management` image from Docker Hub — publicly available, no authentication required.

### GitHub Actions pipeline

The workflow in `.github/workflows/ci.yml` runs on every push to `main` and on every pull request:

| Step | Job | Details |
|---|---|---|
| 1 | Lint | `flake8` on `frontend/app.py` and `worker/worker.py` (max-line-length 120) |
| 2 | Build & push frontend | `docker/build-push-action` → `ghcr.io/<owner>/skywatch-frontend:<sha7>` |
| 3 | Build & push worker | `docker/build-push-action` → `ghcr.io/<owner>/skywatch-worker:<sha7>` |
| 4 | Update Helm values | mikefarah/yq updates `frontend.image.tag` and `worker.image.tag` in `values.yaml` |
| 5 | Commit tag bump | `git commit` with `[skip ci]` to prevent re-trigger loop |

Each image is tagged with the 7-character Git commit SHA (`${GITHUB_SHA::7}`) and also with `latest`. ArgoCD detects the `values.yaml` change and rolls out the new tag automatically within ~3 minutes.

> **Note:** `pip install yq` installs a Python wrapper with incompatible syntax. The pipeline downloads the [mikefarah/yq](https://github.com/mikefarah/yq) binary directly via `wget` to ensure `yq -i` in-place editing works correctly.

## Part 3 — GitOps Deployment (Helm + ArgoCD)

### Helm chart

```
helm/skywatch/
├── Chart.yaml                  # No external dependencies
├── values.yaml                 # Image tags, replicas, resources, RabbitMQ auth
└── templates/
    ├── _helpers.tpl
    ├── frontend-deployment.yaml
    ├── frontend-service.yaml   # NodePort 30080
    ├── worker-deployment.yaml
    ├── rabbitmq-secret.yaml    # Holds RABBITMQ_DEFAULT_*, rabbitmq-{username,password}
    └── rabbitmq.yaml           # StatefulSet (1 replica) + ClusterIP Service
```

#### RabbitMQ — migration from Bitnami sub-chart

The original design used the `bitnami/rabbitmq` Helm sub-chart committed to the repo so ArgoCD didn't need external network access. During the project it was discovered that Bitnami had removed **all** images from Docker Hub — both old and new tags return 404, and the new OCI registry requires authentication for anonymous pulls.

**Solution:** replace the sub-chart with inline Helm templates using the official `rabbitmq:3.13-management` image. Key design decisions in the new templates:

- **Single Secret** (`skywatch-rabbitmq`): holds both `RABBITMQ_DEFAULT_*` keys (consumed by the RabbitMQ container via `envFrom`) and `rabbitmq-{username,password}` keys (referenced by frontend/worker via `secretKeyRef`).
- **TCP socket probes** instead of `rabbitmq-diagnostics ping`: the exec probe timed out under memory pressure (the Erlang VM slows when the memory watermark alarm fires); a TCP check on port 5672 is instant and unaffected by VM load.
- **`ignoreDifferences` for volumeClaimTemplates**: the Kubernetes API server adds default fields (`storageClass`, `volumeMode`) to the VolumeClaimTemplate that aren't in the manifest, causing ArgoCD to report perpetual OutOfSync. Fixed via the Application's `ignoreDifferences` stanza.

### ArgoCD

| Setting | Value |
|---|---|
| Version | ArgoCD v2.11.2 |
| Namespace | `argocd` |
| Exposure | NodePort 30081 on master |
| Node placement | All 7 ArgoCD pods pinned to `skywatch-worker` via `nodeSelector` |
| Sync policy | Automated — `prune: true`, `selfHeal: true` |
| Sync options | `CreateNamespace=true`, `ServerSideApply=true` |
| Source | `github.com/yfreifeld/skywatch` · path `helm/skywatch` · branch `main` |

`argocd/application.yaml`:

```yaml
spec:
  ignoreDifferences:
    - group: apps
      kind: StatefulSet
      name: skywatch-rabbitmq
      jsonPointers:
        - /spec/volumeClaimTemplates

  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
      - ServerSideApply=true
```

### Memory management on t3.micro

Each node has 913 MB RAM. Running K3s, ArgoCD, and application pods together saturated memory. Mitigations applied:

- 2 GB swap file on master and worker2 (`fallocate -l 2G /swapfile`)
- All ArgoCD pods moved to worker1 via `nodeSelector`; skywatch app moved to worker2
- Master tainted `node-role.kubernetes.io/control-plane:NoSchedule` — no app workloads land there
- Third node (worker2) added via Terraform to give the app its own dedicated node

## Part 4 — Observability (Prometheus + Grafana)

### Installation

The `kube-prometheus-stack` Helm chart ships large CRDs that time out on the t3.micro API server when installed via `helm install`. The solution is to apply the CRDs with `kubectl apply --server-side` first, then install the chart with `--skip-crds`:

```bash
# Step 1 — CRDs (applied individually to avoid timeouts)
BASE=https://raw.githubusercontent.com/prometheus-community/helm-charts/main/charts/kube-prometheus-stack/charts/crds/crds
for crd in crd-alertmanagerconfigs crd-alertmanagers crd-podmonitors crd-probes \
           crd-prometheusagents crd-prometheuses crd-prometheusrules \
           crd-scrapeconfigs crd-servicemonitors crd-thanosrulers; do
  kubectl apply --server-side --request-timeout=3m -f "${BASE}/${crd}.yaml"
  sleep 10
done

# Step 2 — Chart (skip CRDs already applied above)
helm install kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace \
  --skip-crds \
  -f helm/monitoring/values.yaml
```

### Slim values for t3.micro (`helm/monitoring/values.yaml`)

```yaml
alertmanager:
  enabled: false        # saves ~100 MB RAM

prometheus:
  prometheusSpec:
    retention: 24h
    nodeSelector:
      kubernetes.io/hostname: skywatch-worker2
    resources:
      limits: { cpu: 500m, memory: 512Mi }

grafana:
  nodeSelector:
    kubernetes.io/hostname: skywatch-worker2
  service:
    type: NodePort
    nodePort: 30030
  adminPassword: "skywatch-grafana"
  resources:
    limits: { cpu: 200m, memory: 128Mi }
```

### RabbitMQ ServiceMonitor

The `rabbitmq:3.13-management` image starts a Prometheus endpoint on port 15692 automatically (the `rabbitmq_prometheus` plugin is enabled by default). A ServiceMonitor tells Prometheus to scrape it:

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: skywatch-rabbitmq
  namespace: skywatch
spec:
  selector:
    matchLabels:
      app.kubernetes.io/name: skywatch-rabbitmq
  endpoints:
    - port: prometheus
      interval: 30s
```

| Service | URL |
|---|---|
| SkyWatch app | `http://<worker2-ip>:30080` |
| ArgoCD | `https://<master-ip>:30081` (admin / see secret) |

## Lessons Learned

| # | Finding | Takeaway |
|---|---|---|
| 1 | K3s v1.35 OOM on t3.micro | Always pin K3s to a known-good version on resource-constrained nodes. `v1.29.5+k3s1` is stable on 1 GB. |
| 2 | Worker couldn't join via master public IP | In AWS, intra-VPC traffic must use private IPs. Public IPs egress via IGW and bypass SG self-rules. |
| 3 | `kubectl` TLS cert error from laptop | `--tls-san` must include the public IP at install time, not just the hostname. |
| 4 | K3s token not stored in agent env file | The K3s installer reads `K3S_TOKEN` from the environment, not from `--token` CLI flag. |
| 5 | `ansible-playbook --limit worker` broke cluster | `--limit` skips the master play so `k3s_token` fact is never set. Always run the full playbook. |
| 6 | Bitnami images removed from Docker Hub | Vendor image registries can disappear. Use official Docker Hub images or mirror to your own registry. |
| 7 | `pip install yq` vs mikefarah yq | Two incompatible tools share the same name. Always download the mikefarah binary explicitly in CI. |
| 8 | ArgoCD perpetual OutOfSync for StatefulSet | API server adds default fields to `volumeClaimTemplates`. Use `ignoreDifferences` in the ArgoCD Application. |
| 9 | Re-running Ansible wiped live cluster | Shell tasks with `removes:` guard are not idempotent when the uninstall script exists on a healthy node. Add an active-service check first. |
| 10 | Prometheus CRD install times out via Helm | Apply large CRDs with `kubectl apply --server-side` first, then `helm install --skip-crds`. |

## Session Quick-Start

Run these commands at the start of each session to spin up the cluster from scratch:

```bash
# 1. Provision infrastructure (~2 min)
cd terraform && terraform apply -auto-approve

# 2. Install K3s + join all workers (~3 min)
cd ../ansible && ansible-playbook -i inventory.ini playbook.yml

# 3. Point kubectl at the new cluster
export KUBECONFIG=$(pwd)/../kubeconfig

# 4. Add swap to worker2 (needed for Prometheus)
ssh -i ~/.ssh/skywatch-key.pem ubuntu@<worker2-ip> \
  "sudo fallocate -l 2G /swapfile && sudo chmod 600 /swapfile && \
   sudo mkswap /swapfile && sudo swapon /swapfile"

# 5. Install ArgoCD and pin to worker1
kubectl create namespace argocd
kubectl apply -n argocd \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/v2.11.2/manifests/install.yaml
kubectl patch svc argocd-server -n argocd \
  -p '{"spec":{"type":"NodePort","ports":[{"port":443,"targetPort":8080,"nodePort":30081}]}}'
NODE_SELECTOR='{"spec":{"template":{"spec":{"nodeSelector":{"kubernetes.io/hostname":"skywatch-worker"}}}}}'
for deploy in argocd-server argocd-repo-server argocd-applicationset-controller \
              argocd-dex-server argocd-notifications-controller argocd-redis; do
  kubectl patch deployment $deploy -n argocd -p "$NODE_SELECTOR"
done
kubectl patch statefulset argocd-application-controller -n argocd -p "$NODE_SELECTOR"

# 6. Deploy skywatch via ArgoCD (auto-syncs from Git)
kubectl apply -f argocd/application.yaml

# 7. Taint master so no app pods land there
kubectl taint node skywatch-master node-role.kubernetes.io/control-plane:NoSchedule

# 8. Destroy at end of session (stay in free tier)
cd terraform && terraform destroy -auto-approve
```
